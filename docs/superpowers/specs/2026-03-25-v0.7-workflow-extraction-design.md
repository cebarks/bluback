# v0.7 Workflow Extraction Design (Item 10)

Extract duplicated business logic from `cli.rs` and TUI modules into a shared `workflow.rs` module. No trait abstraction — just shared pure functions that both CLI and TUI call directly.

---

## Motivation

~150 lines of business logic are duplicated between CLI (`src/cli.rs`) and TUI (`src/tui/wizard.rs`, `src/tui/dashboard.rs`). The duplication covers filename generation, remux job setup, and overwrite handling. Extracting these into shared functions eliminates the duplication, makes the logic independently testable, and prepares the codebase for a future GUI frontend.

## What does NOT change

- CLI interactive prompting, headless logic, progress rendering
- TUI state machine, event routing, background threading, rendering
- No `WorkflowUI` trait — the TUI's async state machine and CLI's sequential model are fundamentally different execution models that don't benefit from a shared trait

---

## Extracted Functions

### 1. `build_output_filename()`

Unified filename generation for a single playlist. Replaces the per-playlist closure in `cli::build_filenames()` (~50 lines) and `tui::wizard::playlist_filename()` (~114 lines).

**Signature:**
```rust
pub fn build_output_filename(
    playlist: &Playlist,
    episodes: &[Episode],
    season: u32,
    is_movie: bool,
    is_special: bool,
    movie_title: Option<(&str, &str)>,   // (title, year)
    show_name: &str,
    label: &str,
    label_info: Option<&LabelInfo>,
    config: &Config,
    cli_format: Option<&str>,
    cli_preset: Option<&str>,
    media_info: Option<&MediaInfo>,
    part: Option<u32>,                    // multi-part movie index
) -> String
```

**Logic:**
1. Build `extra_vars` HashMap: show, disc, label, playlist (using `playlist.num`)
2. Resolve format template internally — the function owns the `use_custom_format` decision:
   - Compute `use_custom_format = cli_format.is_some() || cli_preset.is_some() || config.tv_format.is_some() || config.movie_format.is_some() || config.preset.is_some()`
   - If custom format: resolve via `config.resolve_format()` / `config.resolve_special_format()` and pass `Some(template)` to `make_filename`
   - If no custom format: pass `None` to `make_filename` (uses legacy default)
   - This resolves a minor inconsistency between CLI and TUI where the `use_custom_format` check differed slightly — the canonical logic above is authoritative
3. Route to movie/special/TV path:
   - **Movie:** call `make_movie_filename()` with title, year, part
   - **Special:** always uses custom format (resolve via `config.resolve_special_format(cli_format)`), call `make_filename()` with actual season
   - **TV:** call `make_filename()` with resolved format (or None for default)
4. Return the generated filename string

**Caller responsibility for specials:** The `episodes` parameter must be pre-built by the caller. For TUI, specials episodes are already assigned interactively in the Playlist Manager. For CLI, the caller must build the special `Episode` vector (with the sequential SP counter) before calling this function — the counter state lives in the caller's loop, not in this function.

**Callers after extraction:**
- `cli::build_filenames()` reduces to a loop calling `build_output_filename()` per playlist
- `tui::wizard::playlist_filename()` reduces to a thin wrapper calling `build_output_filename()`

### 2. `check_overwrite()`

Decides what to do when an output file already exists.

**Signature:**
```rust
pub enum OverwriteAction {
    Proceed,
    Skip(u64),              // file size of existing file
    DeleteAndProceed(u64),  // file size before deletion
}

pub fn check_overwrite(output: &Path, overwrite: bool) -> std::io::Result<OverwriteAction>
```

**Logic:**
1. If file doesn't exist → `Proceed`
2. If file exists and `overwrite` is false → `Skip(size)`
3. If file exists and `overwrite` is true → delete file, return `DeleteAndProceed(size)`

**Callers:** Both `cli::rip_selected()` and `tui::dashboard::start_next_job()` currently inline this logic (~15 lines each). After extraction, they call `check_overwrite()` and handle the result in their UI-specific way (CLI prints message, TUI sets `PlaylistStatus::Skipped`).

### 3. `prepare_remux_options()`

Constructs `RemuxOptions` for a single playlist rip. Consolidates chapter extraction and option building.

**Signature:**
```rust
pub fn prepare_remux_options(
    device: &str,
    playlist: &Playlist,
    output: &Path,
    mount_point: Option<&str>,
    stream_selection: StreamSelection,
    reserve_index_space_kb: u32,
) -> RemuxOptions
```

**Signature (final):**
```rust
pub fn prepare_remux_options(
    device: &str,
    playlist: &Playlist,
    output: &Path,
    mount_point: Option<&str>,
    stream_selection: StreamSelection,
    cancel: Arc<AtomicBool>,
    reserve_index_space_kb: u32,
) -> RemuxOptions
```

**Logic:**
1. Extract chapters from mounted disc (if `mount_point` is `Some`). Chapter extraction failures are swallowed — returns empty chapters on error, matching current caller behavior (both use `.unwrap_or_default()`).
2. Build and return `RemuxOptions` struct with the provided cancel flag

The `cancel` flag is caller-provided since cancellation is UI-specific (CLI uses global `CANCELLED`, TUI uses per-job cancel flags).

### ~~4. `detect_movie_mode()`~~ — Dropped

Movie mode detection is a single boolean expression (`movie_flag || (playlist_count == 1 && season.is_none())`). Extracting a one-liner into a separate module adds indirection without meaningful deduplication. Both callers keep their inline expression.

## Files Changed

| File | Change |
|------|--------|
| `src/workflow.rs` | New module — 3 functions + `OverwriteAction` enum |
| `src/main.rs` | Add `mod workflow;` |
| `src/cli.rs` | Replace duplicated logic with calls to `workflow::*` |
| `src/tui/wizard.rs` | Replace `playlist_filename()` internals with `workflow::build_output_filename()` |
| `src/tui/dashboard.rs` | Replace remux setup with `workflow::prepare_remux_options()` + `workflow::check_overwrite()` |

## Dependencies

No new crate dependencies. `workflow.rs` depends on existing types from `types.rs`, `config.rs`, `util.rs`, `chapters.rs`, and `media/remux.rs`.

## Testing

Unit tests for all 3 functions in `workflow.rs`:

- `build_output_filename()`: TV format, movie format, special format, with/without media info, multi-part movie, default (no custom format)
- `check_overwrite()`: file doesn't exist, exists + no overwrite, exists + overwrite (uses temp dir)
- `prepare_remux_options()`: with/without mount point — chapter extraction is tested via existing `chapters.rs` tests; here we test that the function returns valid `RemuxOptions` and swallows chapter errors gracefully

All functions are pure or near-pure, testable without hardware.
