# v0.7 CLI Features Design (Items 11-14)

Design spec for roadmap items 11-14: specials CLI parity, headless progress, verbose playlist listing, and setup validation.

---

## Item 11: `--specials` CLI Flag

### Interface

```
--specials <SEL>    Mark playlists as specials (S00), e.g. 4,5 or 4-5
```

- Uses **filtered index**, consistent with `--playlists`
- Mutually exclusive with `--movie` (clap conflict group)
- Parsed by existing `parse_selection()` from `util.rs`

### Behavior

- Specials playlists are auto-assigned S{season}SP01, S{season}SP02, etc. (sequential, using the actual season number)
- Non-specials playlists keep normal sequential episode assignment (unaffected)
- Works in both interactive and headless CLI mode
- Interactive CLI: specials are pre-marked before the review prompt, user can still customize filenames
- Headless CLI: specials applied automatically, no prompt
- `--specials` without `--playlists`: works naturally — all filtered playlists are selected by default, specials indices resolve against that full filtered list
- Validation: specials indices must be a subset of selected playlists — warn and skip invalid indices

### Filename generation

- Specials use `config.resolve_special_format()` with the **actual season number** (not season 0)
- Non-specials use the normal TV format with the configured season number
- `DEFAULT_SPECIAL_FORMAT` changes from `"{show} S00E{episode} {title}.mkv"` to `"{show} S{season}SP{episode} {title}.mkv"`. The `{episode}` placeholder in the special format refers to the special's own sequential counter (SP01, SP02, ...), not the regular episode numbering.
- Episode numbering: specials are removed from the regular episode sequence. Regular episodes are renumbered contiguously, and specials get their own independent SP counter. This matches TUI behavior where the `s` toggle removes the playlist from regular assignment.
- `--start-episode` interaction: the start episode applies to the regular (non-special) sequence only. Specials always start at SP01.

### Examples

```bash
# Rip playlists 1-7, with 6 and 7 as specials
bluback --playlists 1-7 --specials 6,7 --season 2

# Produces:
#   S02E01_Title.mkv through S02E05_Title.mkv (playlists 1-5)
#   S02SP01_Title.mkv and S02SP02_Title.mkv (playlists 6-7)

# With --start-episode and specials in the middle
bluback --playlists 1-7 --specials 3,5 --season 2 --start-episode 4

# Regular sequence (playlists 1,2,4,6,7): S02E04 through S02E08
# Specials (playlists 3,5): S02SP01, S02SP02
```

### Implementation

- **`src/main.rs`**: Add `specials: Option<String>` to `Args` struct with clap conflict `movie`
- **`src/cli.rs`**: Specials parsing happens in `cli::run()`, alongside playlist selection (after filtering, before `build_filenames()`). Convert filtered indices to playlist numbers, producing a `HashSet<String>` (same type as TUI's `wizard.specials`). Pass this set into `build_filenames()`, which checks membership to route each playlist to either `resolve_special_format()` with the actual season number or the normal TV format.
- **`src/util.rs`**: No changes needed — `parse_selection()` already handles the format

### Testing

- Unit tests: specials selection parsing, filename generation with mixed regular + specials, specials indices out of range warning
- Integration: combined `--playlists` + `--specials` flag interaction

### TUI changes

- **Naming scheme change**: TUI specials currently pass season 0 to `make_filename()`. Update to pass the actual season number, matching the new `S{season}SP{episode}` format.
- **Display**: The `[S]` marker in the Playlist Manager should show the new format (e.g., `SP01` instead of `S00E01`).
- **`DEFAULT_SPECIAL_FORMAT`**: Updated in `config.rs` — affects both TUI and CLI.
- No batch marking changes. Single-row `s` toggle is sufficient.

## Item 12: Headless Progress Output

### Problem

Current CLI progress uses `\r` carriage return to overwrite the line in-place. When stdout is piped to a file or another process (non-TTY), those `\r` lines stack up as garbage.

### Design

Two progress paths based on TTY detection:

- **TTY stdout**: Keep existing `\r` carriage return behavior (high-frequency, smooth in-place updates)
- **Non-TTY stdout**: Print newline-terminated progress lines at 10-second wall-clock intervals

### Non-TTY format

```
[00001] 12% 245MiB 42MiB/s ETA 3:22
[00001] 24% 510MiB 43MiB/s ETA 2:55
[00001] 100% 2.1GiB — done
```

- Playlist number in brackets for multi-playlist disambiguation
- Percentage from `out_time_secs / total_seconds`
- Completion line printed at 100% regardless of interval

### Implementation

- **`src/cli.rs`**: In the progress callback closure, use the existing `atty_stdout()` helper (which uses `std::io::IsTerminal`) to choose the path:
  - TTY: existing `\r` path unchanged
  - Non-TTY: track `last_progress_print: Instant` (captured by closure). On each callback, if `elapsed() >= 10s`, print with `println!` and reset timer
- TTY check should happen once before creating the closure and be captured as a `bool`, not checked per-callback
- Print a start line when remux begins and a completion line when it finishes, regardless of interval
- For very short playlists (< 10s total), at minimum a start and completion line are printed

### Testing

- Unit test for the non-TTY progress line formatting function
- No hardware-dependent tests

## Item 13: `--list-playlists` Verbose Stream Info

### Interface

```
--verbose, -v    Show detailed stream info (with --list-playlists)
```

General-purpose flag. Currently only affects `--list-playlists` output, but available for future use.

### Default output (unchanged)

```
#  Playlist  Duration  Ch  Sel
1  00001     1:23:45   20   1
2  00002     0:45:30   15   2
```

### Verbose output

Adds video and audio columns from per-playlist FFmpeg probe:

```
#  Playlist  Duration  Ch  Video              Audio                      Sel
1  00001     1:23:45   20  H.264 1080p 23.97  DTS-HD MA 5.1, AC3 2.0     1
2  00002     0:45:30   15  H.264 1080p 23.97  DTS-HD MA 5.1, AC3 2.0     2
```

- **Video column**: codec + resolution + framerate (from `MediaInfo`)
- **Audio column**: comma-separated list of codec + channel layout per audio stream
- Video info comes from `probe_media_info()` (codec, resolution, framerate). Audio stream list comes from `probe_streams()` which returns `StreamInfo.audio_streams: Vec<AudioStream>` — this is needed because `probe_media_info()` only returns the first audio stream. Both functions open the device, so call `probe_streams()` once and extract video info from `MediaInfo` and audio list from `StreamInfo`.

### Performance

Each probe opens the disc via FFmpeg + libbluray. For a typical disc with 5-10 episode playlists, this adds a few seconds total. Acceptable for an explicit `--verbose` request.

Print `"Probing streams..."` to stderr before per-playlist probes so the user knows why it's slower.

### Implementation

- **`src/main.rs`**: Add `verbose: bool` (short `-v`) to `Args` struct
- **`src/cli.rs`**: In `list_playlists()`, when verbose:
  1. Print `"Probing streams..."` to stderr
  2. Call `probe_media_info()` and `probe_streams()` per playlist (or a combined call)
  3. Format video/audio info into additional columns
- **Formatting**: Extract a helper function for the video/audio column strings, testable independently

### Testing

- Unit test for video/audio column formatting
- Existing `probe_media_info` and formatter helpers already tested

## Item 14: `--check` Setup Validation

### Interface

```
--check    Validate environment and configuration, then exit
```

No disc required. Dispatched in `main.rs` after config loading/validation but before `aacs::preflight()`. The check module calls `find_library()` and `command_exists()` directly to inspect the environment non-destructively — it does NOT go through the AACS preflight flow (which sets env vars and can bail on errors).

### Checks

| # | Check | Type | Method |
|---|-------|------|--------|
| 1 | FFmpeg libraries | Required | `ffmpeg::init()`, report version if available |
| 2 | libbluray | Required | `find_library("libbluray")` via ldconfig |
| 3 | libaacs | Required | `find_library("libaacs")` via ldconfig |
| 4 | KEYDB.cfg | Required | File exists at `~/.config/aacs/KEYDB.cfg` |
| 5 | libmmbd | Optional | `find_library("libmmbd")` via ldconfig |
| 6 | makemkvcon | Optional | `command_exists("makemkvcon")` |
| 7 | udisksctl | Required | `command_exists("udisksctl")` |
| 8 | Optical drives | Optional | `detect_optical_drives()`, list found devices |
| 9 | Drive permissions | Optional | `File::open()` read access on detected drives (skipped if no drives) |
| 10 | Output directory | Required | Configured/default dir exists and is writable |
| 11 | TMDb API key | Optional | Config -> flat file -> env var chain |
| 12 | Config file | Optional | `validate_raw_toml()` + `validate_config()` |

### Status labels

```
[PASS] FFmpeg libraries (libavformat 61.1.100)
[PASS] libbluray (0.9.3)
[PASS] libaacs (0.11.1)
[PASS] KEYDB.cfg (~/.config/aacs/KEYDB.cfg)
[MISS] libmmbd (optional — needed for aacs_backend=libmmbd)
[MISS] makemkvcon (optional — needed for aacs_backend=libmmbd)
[PASS] udisksctl
[PASS] Optical drive (/dev/sr0)
[PASS] Drive permissions (/dev/sr0 — readable)
[PASS] Output directory (. — writable)
[WARN] TMDb API key (not configured — TMDb lookup will be unavailable)
[PASS] Config file (~/.config/bluback/config.toml — valid)
```

- `PASS` — present and valid
- `FAIL` — required and missing/broken
- `WARN` — present but degraded (e.g., config has unknown keys, or feature will be limited)
- `MISS` — optional and not found

### Exit code

- `0` if all required checks pass (optional MISS/WARN are fine)
- `2` (usage/config error) if any required check fails

### Implementation

- **`src/main.rs`**: Add `check: bool` to `Args` struct. Dispatch after config load but before `aacs::preflight()`.
- **New `src/check.rs`**: `pub fn run_check(config: &Config, args: &Args) -> i32`
  - Runs all checks sequentially, collects results
  - Prints formatted output
  - Returns exit code
- **`src/aacs.rs`**: Make `find_library()` and `command_exists()` `pub` (currently module-private)
- **`src/disc.rs`**: `detect_optical_drives()` already `pub`

### Drive permissions check

When optical drives are detected, attempt `std::fs::File::open()` on each. If permission denied, report `FAIL` with a hint about group membership:

```
[FAIL] Drive permissions (/dev/sr0 — permission denied, check cdrom/optical group)
```

If no drives detected, skip the permissions check (the "Optical drives" line already shows the failure).

### Config check details

- If no config file exists: `[PASS] Config file (no config file — using defaults)`
- If config exists and valid: `[PASS] Config file (~/.config/bluback/config.toml — valid)`
- If config has warnings: `[WARN] Config file (unknown keys: foo, bar)`
- If config can't be parsed: `[FAIL] Config file (parse error: ...)`

### Testing

- Unit tests for check result formatting and status classification
- Individual check functions already tested in their respective modules

## Dependencies

No new crate dependencies. All functionality built on existing helpers and stdlib.

## Interactions between items

- `--specials` + `--playlists`: specials must be subset of selected playlists
- `--specials` + `--movie`: mutually exclusive (clap conflict)
- `--verbose` + `--list-playlists`: verbose only has effect with list-playlists (no error otherwise, just ignored)
- `--check`: independent of all other flags, exits before any disc/rip logic
