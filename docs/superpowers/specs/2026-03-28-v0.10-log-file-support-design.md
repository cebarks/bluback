# v0.10 Log File Support — Design Spec

## Overview

Add structured file logging to bluback using the `log` crate facade with `fern` as the dispatch backend. Every session automatically writes a log file for post-mortem debugging and audit trail purposes. Replaces scattered `eprintln!` calls with unified `log::warn!`/`log::error!` macros and adds new `log::info!`/`log::debug!` calls at workflow milestones.

## Architecture

### Crate Selection

- **`log`** — facade crate (lightweight, no transitive deps). All source files use `log::info!`, `log::warn!`, etc.
- **`fern`** — dispatch backend (~400 lines, no transitive deps beyond `log`). Configures multi-output routing with per-output level filtering.

No async runtime, no `tracing` — this is a synchronous blocking app and `log` is the right weight class.

### Output Routing

Two output targets configured via `fern::Dispatch`:

| Output | When | Default Level Filter | Content |
|---|---|---|---|
| **Log file** | Always (unless `--no-log`) | `debug` | All log events: session info, disc detection, rip progress, warnings, errors, FFmpeg callback output |
| **stderr** (CLI mode only) | CLI mode, never in TUI mode | `warn` (configurable via `--log-level`) | Warnings and errors echoed to terminal |

TUI mode disables stderr output entirely — ratatui owns the terminal. All logging goes to file only.

The stderr level is configurable: `--log-level debug` makes debug+ messages appear on stderr in CLI mode, effectively acting as a verbose/debug mode for the terminal. This does not change the file level.

`fern` setup (conceptual):

```rust
fern::Dispatch::new()
    .level(LevelFilter::Debug)
    .chain(file_dispatch)           // always: debug+ to file
    .chain(
        fern::Dispatch::new()
            .level(cli_stderr_level) // configurable, default Warn
            .chain(io::stderr())     // CLI mode only
    )
    .apply()?;
```

### Log Format

Human-readable, timestamped plain text:

```
2026-03-28 14:32:01 [INFO] Session started: device=/dev/sr0, output=./
2026-03-28 14:32:03 [INFO] Disc detected: BREAKING_BAD_S1D1
2026-03-28 14:32:05 [DEBUG] Found 12 playlists, 4 above min_duration (900s)
2026-03-28 14:32:05 [WARN] AACS: system libaacs.so is a symlink to libmmbd
2026-03-28 14:32:10 [DEBUG] [ffmpeg] Opening bluray:///dev/sr0 playlist 00800
```

Target-prefixed lines (e.g., `[ffmpeg]`) for messages from FFmpeg's av_log callback.

### Session Header

Each log file starts with a structured preamble for quick audit context:

```
=== bluback 0.10.0 ===
Started: 2026-03-28 14:32:01
Platform: linux x86_64
Device: /dev/sr0
Output: /export/media/tv/
Config: ~/.config/bluback/config.toml
AACS backend: libaacs
===
```

## Log Files

### Location & Naming

- **Directory:** `~/.local/share/bluback/logs/` (XDG_DATA_HOME compliant)
- **Filename:** `bluback-2026-03-28T14-32-01.log` (ISO 8601 with `-` replacing `:` for filesystem safety)
- **Created:** On startup, after config loading but before AACS preflight

### Rotation

On startup, before creating the new log file:

1. List `*.log` files in the log directory
2. Sort by filename (chronological due to timestamp prefix)
3. Delete oldest files if count exceeds `max_log_files` (default: 10)

No size-based rotation — Blu-ray rips are infrequent and individual log files are small (typically <100 KB).

### Disabling

`--no-log` flag or `log_file = false` in config disables file output. Stderr output in CLI mode still works (controlled separately by `--log-level`).

## CLI Flags & Config Options

### New CLI Flags

```
--log-level <LEVEL>    Stderr verbosity: error, warn, info, debug, trace [default: warn]
--no-log               Disable log file output
--log-file <PATH>      Custom log file path (overrides default location)
```

`--no-log` takes precedence over `--log-file` if both are specified.

### New Config Options

```toml
log_file = true                                  # Enable/disable file logging (default: true)
log_level = "warn"                               # Stderr verbosity in CLI mode (default: "warn")
log_dir = "~/.local/share/bluback/logs"          # Log directory (default: XDG_DATA_HOME)
max_log_files = 10                               # Rotation: keep last N log files (default: 10)
```

### Priority Chain

Matches existing config patterns: CLI flag → env var (`BLUBACK_LOG_LEVEL`, `BLUBACK_LOG_DIR`) → config file → default.

### Settings Panel

Add a "Logging" separator with:
- `log_level` — Choice variant (error/warn/info/debug/trace)
- `log_file` — Toggle variant

`log_dir` and `max_log_files` are config-file-only — not worth cluttering the settings UI.

## Module Structure

### New: `src/logging.rs` (~100-150 lines)

```rust
/// Initialize logging: create log directory, rotate old logs, configure fern dispatch.
/// Returns the log file path (None if logging disabled).
pub fn init(config: &Config, log_level: LevelFilter, log_file_path: Option<PathBuf>, no_log: bool, is_tui: bool) -> anyhow::Result<Option<PathBuf>>

/// Delete oldest log files beyond max_files limit.
fn rotate_logs(log_dir: &Path, max_files: usize) -> anyhow::Result<()>

/// Build session header string.
fn session_header(version: &str, device: &str, output_dir: &str, config_path: &str, aacs_backend: &str) -> String
```

Called from `main.rs` early in `run()`, after config loading but before AACS preflight — so preflight warnings are captured.

### Config Additions (`src/config.rs`)

- 4 new fields: `log_file`, `log_level`, `log_dir`, `max_log_files`
- Parsing + defaults
- Validation: `max_log_files > 0`, valid log level string
- `KNOWN_KEYS` additions
- `BLUBACK_LOG_LEVEL` and `BLUBACK_LOG_DIR` env var support

### `--check` Addition

Add a "Log directory" check: verify directory is writable, report current log file count.

## Sensitive Data

**Never log the TMDb API key.** It can appear in several places:

- **TMDb request URLs** — `api.themoviedb.org/3/search/tv?api_key=XXXX`. Log the request path/action but strip or omit the key: `log::info!("TMDb search: query={}", query)`, not the full URL.
- **Config values** — When logging config resolution at debug level, skip `tmdb_api_key`. Log its *source* (config file / env var / flat file) but not the value.
- **Session header** — Do not include the API key. The header already omits it (lists AACS backend, not TMDb key).
- **Settings panel logging** — If logging setting changes, mask the API key field.

General rule: treat `tmdb_api_key` the same way the settings panel does (masked display). No other config fields are sensitive.

## FFmpeg Callback Integration

The existing thread-local `THREAD_LOG_BUFFER` capture in `media/probe.rs` stays — it serves the specific purpose of parsing playlist info from libbluray output via regex.

After pushing to the thread-local buffer, captured lines are also routed through the `log` crate:

```rust
log::debug!(target: "ffmpeg", "{}", line.trim());
```

The `target: "ffmpeg"` prefix appears in log file output as `[ffmpeg]`, making FFmpeg-originated messages easy to filter.

## Migration Strategy

Systematic replacement by category, not big-bang:

1. **`eprintln!("Warning: ...")`** → `log::warn!(...)` — aacs.rs, main.rs, media/remux.rs (~10 calls)
2. **`eprintln!` in error paths** → `log::error!(...)` — main.rs, coordinator.rs (~5 calls)
3. **New `log::info!` at workflow milestones** — disc detected, playlists scanned, TMDb result, rip started, rip completed/failed, session config summary (~15-20 new calls across cli.rs, session.rs, disc.rs, media/probe.rs, media/remux.rs, workflow.rs)
4. **New `log::debug!` for internals** — config resolution, playlist filtering decisions, filename generation, chapter extraction details (~10-15 new calls)
5. **FFmpeg callback** — add `log::debug!(target: "ffmpeg", ...)` after thread-local buffer push (media/probe.rs, 1 call in the macro)
6. **Leave `println!` alone** — interactive prompts, `--list-playlists` output, `--check` output, progress `\r` updates are user-facing, not logging

## Testing

### Unit Tests (in `logging.rs`)

- `rotate_logs`: deletes oldest when > max, no-op when under limit, handles empty directory, handles missing directory
- `session_header`: format correctness
- Log level string parsing

### Config Tests

- New fields parse correctly from TOML
- Validation rejects `max_log_files = 0`
- Unknown key detection includes new keys in `KNOWN_KEYS`

### Manual Testing

- Run a rip, verify log file exists at expected path with expected content
- Verify stderr output respects `--log-level` in CLI mode
- Verify no stderr output in TUI mode
- Verify `--no-log` suppresses file creation
- Verify rotation deletes oldest files

No integration test needed for `fern` itself — we trust the crate.

## Out of Scope

- **Log viewer TUI screen** — `less`/`rg` are fine for reading logs
- **Remote/network logging** — not applicable for a CLI tool
- **Per-rip log files** — single session log is sufficient; disc history (roadmap item 31) handles per-rip records
- **JSON log format** — human-readable only; no `--log-format` option
