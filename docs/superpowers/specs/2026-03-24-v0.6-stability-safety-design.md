# v0.6 Design Spec: Stability & Safety Foundation

## Summary

v0.6 addresses stability issues, error handling gaps, and safety problems in bluback's core pipeline. Ten items across six PRs, ordered by dependency.

## PR 1: Quick Safety Fixes

### 1.1 Fix `detect_optical_drives()` panic

**Problem:** `main.rs:157` â€” `drives[0].clone()` panics on empty vec. The function currently falls back to `/dev/sr0` when lsblk finds no `rom` devices, masking the real problem: there's no optical drive.

**Fix:** Remove the `/dev/sr0` fallback from `detect_optical_drives()`. Return the raw `Vec<PathBuf>`. In `main()`, handle empty with `anyhow::bail!("No optical drives detected")`.

**Files:** `src/disc.rs:38-40`, `src/main.rs:155-158`

**Tests:** Unit test for `detect_optical_drives()` isn't practical (requires lsblk), but the empty-vec handling in main is straightforward enough to verify by inspection.

### 1.2 TMDb request timeout

**Problem:** `tmdb.rs:50-52` â€” `ureq::get().call()` uses ureq's implicit 30s default. Not explicitly documented, and 30s is too long for a metadata lookup.

**Fix:** ureq 3.x configures timeouts at the agent level, not per-request. Create a module-level `Agent` with `timeout_global(Some(Duration::from_secs(15)))` and use it for all TMDb calls instead of bare `ureq::get()`.

```rust
use std::sync::LazyLock;
static TMDB_AGENT: LazyLock<ureq::Agent> = LazyLock::new(|| {
    ureq::Agent::config_builder()
        .timeout_global(Some(Duration::from_secs(15)))
        .build()
        .into()
});
```

Replace `ureq::get(&url)` with `TMDB_AGENT.get(&url)` in `tmdb_get()`.

**Files:** `src/tmdb.rs:50`

### 1.3 Output directory error propagation (TUI)

**Problem:** `tui/dashboard.rs:322` â€” `create_dir_all(parent).ok()` silently swallows permission errors. If the output directory can't be created, the rip silently fails later with an obscure FFmpeg error.

**Fix:** Replace `.ok()` with proper error handling. If `create_dir_all` fails, set the job status to `PlaylistStatus::Failed` with a clear message ("Failed to create output directory: {path}: {error}") and skip the rip.

**Files:** `src/tui/dashboard.rs:321-323`

**Note:** This item satisfies roadmap item 8 ("Output directory auto-creation"). The auto-creation logic (`create_dir_all`) already exists in both CLI and TUI paths. The only fix needed is propagating errors in the TUI path instead of silently swallowing them.

## PR 2: AACS Backend Detection + libmmbd Support

### 2.1 Background

bluback uses FFmpeg -> libbluray -> libaacs for AACS decryption. libbluray loads the AACS library dynamically with this priority:

1. `LIBAACS_PATH` env var (if set)
2. `libaacs` (standard library resolution)
3. `libmmbd` (MakeMKV's drop-in replacement, fallback)

libmmbd is MakeMKV's LGPL proxy library that spawns `makemkvcon` in the background and delegates decryption to it. When MakeMKV has LibreDrive firmware, libmmbd can decrypt discs that plain libaacs can't (revoked host certs, MKBv72+, AACS 2.0/2.1 UHD).

**Known stability issue:** If `libmmbd.so.0` is installed (e.g., symlinked as `libaacs.so.0`) but `makemkvcon` isn't available, libmmbd's `CMMBDConn::initialize()` blocks trying to spawn the process, causing libbluray to hang. bluback's 60-second timeout (`media/probe.rs:98`) catches this, but the error message is generic.

### 2.2 Config addition

New field on `Config`:

```toml
# AACS decryption backend
# "auto" (default) - let libbluray decide (checks LIBAACS_PATH, then libaacs, then libmmbd)
# "libaacs" - force system libaacs (requires KEYDB.cfg with keys/VUKs)
# "libmmbd" - force MakeMKV's libmmbd (requires makemkvcon in PATH)
aacs_backend = "auto"
```

Accessor: `config.aacs_backend() -> AacsBackend` where `AacsBackend` is an enum: `Auto`, `Libaacs`, `Libmmbd`. Lives in `config.rs`. Default: `Auto`.

CLI flag: `--aacs-backend <auto|libaacs|libmmbd>` â€” overrides config. Added to clap args.

Env var: `BLUBACK_AACS_BACKEND` â€” standard override, integrated into settings panel env var detection.

### 2.3 Preflight logic

New function `aacs::preflight(backend: AacsBackend)` in a new `src/aacs.rs` module. Called from `main()` after config load, before `BD_DEBUG_MASK` setup (line ~127), before any FFmpeg calls.

**Note on `set_var` safety:** Both this preflight and the existing `BD_DEBUG_MASK` setup use `std::env::set_var`, which is `unsafe` in Rust 2024 edition (not thread-safe on Unix). The project is edition 2021 and these calls happen in `main()` before any threads are spawned, so this is safe. If the project migrates to edition 2024, these will require `unsafe` blocks with safety comments.

```
fn preflight(backend: AacsBackend) -> Result<()>
```

Behavior by backend:

**`Libmmbd`:**
1. Check `which makemkvcon` â€” bail if not found: "aacs_backend is set to libmmbd but makemkvcon was not found in PATH. Install MakeMKV or set aacs_backend = \"auto\"."
2. Find libmmbd.so path: check `ldconfig -p | grep libmmbd` first (parses dynamic linker cache, distro-agnostic), fall back to common locations (`/usr/lib64/libmmbd.so.0`, `/usr/lib/x86_64-linux-gnu/libmmbd.so.0`, `/usr/lib/libmmbd.so.0`).
3. Set `LIBAACS_PATH` and `LIBBDPLUS_PATH` env vars to the found path.

**`Libaacs`:**
1. Find the real libaacs.so: check `ldconfig -p | grep libaacs` first, fall back to common locations. Resolve symlinks to ensure it's not actually libmmbd.
2. Set `LIBAACS_PATH` to the real libaacs path.
3. Warn (eprintln) if `~/.config/aacs/KEYDB.cfg` doesn't exist: "Warning: KEYDB.cfg not found at ~/.config/aacs/KEYDB.cfg â€” AACS decryption may fail."

**`Auto`:**
1. Detect if libmmbd is masquerading as libaacs (resolve symlinks on system libaacs.so).
2. If libmmbd is the active backend, check if `makemkvcon` is in PATH.
3. If makemkvcon is not found, warn: "Warning: libmmbd.so is installed as libaacs but makemkvcon was not found. AACS initialization may hang. Consider setting aacs_backend = \"libaacs\" in config."
4. No env var changes â€” let libbluray's natural resolution proceed.

### 2.4 Improved error messages

Update `MediaError::AacsTimeout` display to be context-aware. Add an `aacs_backend: Option<AacsBackend>` field (or a separate function) so the error message can reference the active backend:

| Backend | Timeout message |
|---------|----------------|
| libmmbd / auto+libmmbd detected | "AACS initialization timed out. libmmbd is active but makemkvcon may not be responding. Verify MakeMKV is installed and licensed, or set aacs_backend = \"libaacs\"." |
| libaacs / auto+libaacs | "AACS initialization timed out. This may indicate a drive communication issue." |
| auto, unknown | Current message (generic) |

Update `MediaError::AacsRevoked` to mention libmmbd as an alternative when the backend is libaacs:
"AACS host certificate revoked. This disc requires a per-disc VUK in KEYDB.cfg. If you have MakeMKV with LibreDrive, set aacs_backend = \"libmmbd\"."

### 2.5 Settings panel integration

Add `aacs_backend` as a `SettingItem::Choice` in the settings panel. Options: "auto", "libaacs", "libmmbd". Placed in the device/drive section alongside `device` and `verbose_libbluray`.

Add `BLUBACK_AACS_BACKEND` -> `aacs_backend` mapping to the `ENV_MAPPINGS` array in `types.rs` for env var override support in the settings panel.

### 2.6 Files

- New: `src/aacs.rs` â€” preflight logic, backend detection, library path resolution
- Modified: `src/config.rs` â€” `aacs_backend` field, `AacsBackend` enum, accessor, serialization

- Modified: `src/tui/settings.rs` â€” render/handle new choice item

### 2.7 Tests

- `aacs.rs`: test `find_libmmbd()` and `find_libaacs()` path resolution (mock filesystem not needed â€” test the symlink-resolution logic with known paths)
- `config.rs`: test `aacs_backend` parsing, default, serialization roundtrip
- `aacs.rs`: test preflight error conditions (makemkvcon not in PATH, etc.) â€” these test the logic, not actual library loading

## PR 3: Error Handling & Exit Codes

### 3.1 Unwrap audit

**Scope:** Replace `.unwrap()` with `.expect("context")` in production code paths (outside `#[cfg(test)]` blocks). Grep for `.unwrap()` in all `src/**/*.rs`, filter to production paths only.

Known production unwraps (~12 calls total):

- **Static regex compilation** (`disc.rs:10-11`, `util.rs:50-53`): Safe â€” compile-time constant patterns. Replace with `.expect("valid regex")` where not already using `LazyLock`.
- **Path operations** (`cli.rs:754,785`): `.file_name().unwrap()` â€” safe because paths are constructed via `.join()`. Replace with `.expect("output path has filename")`.
- **Regex captures** (`disc.rs:129-130`): `.parse().unwrap()` on regex-matched digits. Replace with `.expect("regex guarantees numeric capture")`.
- **Settings state lookups** (`types.rs`, `tui/settings.rs`): If any production-path `.unwrap()` calls exist on `.find()`/`.position()` results, replace with `.expect("settings item {key} must exist")`. These are safe (construction guarantees existence) but better panic messages help debugging.

**Approach:** Mechanical replacement â€” no logic changes. Each `.unwrap()` gets an `.expect()` with a brief explanation of why it's safe. The implementer should grep fresh rather than relying on cached line numbers.

**Files:** All `src/**/*.rs` (production paths only)

### 3.2 Structured exit codes

**Codes:**

| Code | Meaning | When |
|------|---------|------|
| 0 | Success | Normal completion |
| 1 | Runtime error | Rip failure, FFmpeg error, I/O error |
| 2 | Usage/config error | Bad CLI args, invalid config, missing dependencies |
| 3 | No disc/device | No optical drive, no disc inserted, device not found |
| 4 | User cancelled | Ctrl+C, `q` during rip |

**Implementation:** `main()` currently returns `anyhow::Result<()>`. Change to:

```rust
fn main() {
    let code = run();
    std::process::exit(code);
}

fn run() -> i32 {
    // current main() body, but map errors to exit codes
}
```

Map `MediaError` variants to codes: `AacsRevoked`/`AacsAuthFailed`/`AacsTimeout` -> 1, `DeviceNotFound`/`NoDisc` -> 3, `Cancelled` -> 4. Config/parse errors -> 2. Everything else -> 1.

TUI mode currently swallows errors internally (shows on Done screen, exits 0). Change: if any rip job failed, exit 1. If user quit during rip, exit 4.

**Files:** `src/main.rs`

### 3.3 Config validation on load

**Goal:** Warn on problems, don't fail (forward-compatible with future config keys).

**Validations:**

- **Unknown keys:** After `toml::from_str()`, re-parse as `toml::Value` and diff keys against known fields. Warn for each unknown key: "Warning: unknown config key '{key}' in {path} (typo?)".
- **Numeric bounds:** `min_duration > 0`, `reserve_index_space` in range 0..10000 (10 MB max).
- **Format template syntax:** Check for unmatched `{` or `}` in format strings.
- **aacs_backend value:** Must be "auto", "libaacs", or "libmmbd" (if present).

**Implementation:** New `config::validate(config: &Config, raw_toml: &str, path: &Path)` function. Called after `load_from()` in `main()`. Prints warnings to stderr. Never returns an error (forward-compat).

**Files:** `src/config.rs`

**Tests:** Test unknown key detection, numeric bounds warnings, format template validation.

## PR 4: Signal Handling + Partial File Cleanup

### 4.1 Signal handling

**Crate:** Use the `ctrlc` crate (simple, well-maintained, 33M downloads). Registers a handler for SIGINT and SIGTERM.

**Handler behavior:**

1. Set a global `AtomicBool` cancel flag (similar to existing cancel mechanism).
2. On first signal: set cancel flag -> remux loop picks it up -> writes trailer -> clean exit with partial file cleanup.
3. On second signal: force exit. The `ctrlc` crate does NOT automatically abort on repeated signals â€” it calls the handler every time. Implement double-signal explicitly: store first signal timestamp in a global `AtomicU64` (epoch millis). If the handler fires again within 2 seconds, call `std::process::exit(130)` from the handler.

```rust
static CANCEL: AtomicBool = AtomicBool::new(false);
static FIRST_SIGNAL: AtomicU64 = AtomicU64::new(0);

ctrlc::set_handler(move || {
    let now = /* epoch millis */;
    let first = FIRST_SIGNAL.load(Ordering::Relaxed);
    if first > 0 && now - first < 2000 {
        // Second signal within 2s â€” force exit
        std::process::exit(130);
    }
    FIRST_SIGNAL.store(now, Ordering::Relaxed);
    CANCEL.store(true, Ordering::Relaxed);
});
```

**Integration point:** `main()` after config load. The cancel flag is the same `Arc<AtomicBool>` already threaded through `RemuxOptions.cancel`. The global `CANCEL` AtomicBool is shared â€” either alias it with the per-session cancel flag or have the main loop check both.

**Files:** `src/main.rs` (handler registration), `Cargo.toml` (add `ctrlc` dep)

### 4.2 Partial file cleanup

**On cancel or remux error:** Delete the incomplete output MKV file.

**Implementation:** After `remux()` returns `Err(MediaError::Cancelled)` or `Err(MediaError::RemuxFailed(_))`, check if the output file exists and delete it. Log the deletion.

**TUI (`tui/dashboard.rs`):** In `poll_active_job`, when a job transitions to `Failed` or `Cancelled`, delete the partial file from the `PlaylistStatus::Ripping` -> `Failed` transition. The output path is `app.args.output.join(&job.filename)`.

**CLI (`cli.rs`):** After `remux()` returns an error, delete the output file if it exists.

**Edge case:** If the signal arrives before `write_header()` (file not yet created), no cleanup needed. If it arrives after trailer is written, the file may be valid but incomplete â€” still delete it (user can re-rip).

### 4.3 Disc unmount on all exit paths

**Current behavior:** `disc.rs` mounts via `udisksctl mount` for chapter extraction, unmounts afterward. But if the process is killed mid-rip, the disc stays mounted.

**Fix:** Track mount state. On exit (normal, error, or signal), unmount if bluback mounted it. Implement via a `MountGuard` struct with both an explicit `cleanup()` method and a `Drop` safety net.

```rust
struct MountGuard {
    device: String,
    mounted_by_us: bool,
}

impl MountGuard {
    /// Explicit cleanup â€” call this before std::process::exit().
    fn cleanup(&mut self) {
        if self.mounted_by_us {
            let _ = unmount_disc(&self.device);
            self.mounted_by_us = false; // prevent double-unmount in Drop
        }
    }
}

impl Drop for MountGuard {
    fn drop(&mut self) {
        self.cleanup();
    }
}
```

**Important:** `std::process::exit()` does NOT run destructors. Since PR 3's exit code design uses `std::process::exit()` in `main()`, the `run()` function must explicitly call `guard.cleanup()` before returning. The `Drop` impl serves as a safety net for normal returns and panics, but the primary cleanup path is explicit. The signal handler's force-exit path (`exit(130)`) will skip cleanup â€” acceptable since it's the "emergency abort" path.

**Files:** `src/disc.rs` (MountGuard), `src/tui/dashboard.rs`, `src/cli.rs` (hold guard)

### 4.4 makemkvcon zombie cleanup

When `aacs_backend` is libmmbd (or auto with libmmbd detected), makemkvcon may leave zombie processes.

**Approach:** On exit, reap zombie child processes. libmmbd spawns makemkvcon as a grandchild (via the FFmpeg worker thread), so `/proc/self/task/*/children` may not find them. Use `waitpid(-1, WNOHANG)` in a loop to reap any zombie children, regardless of origin. This is safe â€” bluback doesn't spawn other long-lived children. Only active when libmmbd backend was detected.

**Implementation:** Part of the explicit cleanup in `run()` before returning (same pattern as MountGuard). Call `aacs::reap_children()` which loops `waitpid(-1, WNOHANG)` until no more zombies.

**Files:** `src/aacs.rs` (cleanup function), `src/main.rs` (call on exit)

## PR 5: Overwrite Protection

### 5.1 Current behavior

Three separate implementations, inconsistent:
- `remux.rs:185-187`: Returns `MediaError::OutputExists` error
- `cli.rs:787-797`: Prints skip message with size, continues to next playlist
- `tui/dashboard.rs:325-330`: Silently marks as Done without ripping

### 5.2 Design

**New config option:**

```toml
# What to do when output file already exists
# false (default) - skip with warning
# true - delete and re-rip
overwrite = false
```

**CLI flag:** `--overwrite` â€” overrides config to true. Added to clap args.

**Env var:** `BLUBACK_OVERWRITE`

Config struct uses `overwrite: Option<bool>` (consistent with all other boolean fields). Accessor: `config.overwrite() -> bool` defaulting to `false`.

**Behavior with `overwrite = false` (default):**
- CLI: Current behavior (print skip message with filename and existing file size). No change.
- TUI: Mark as `PlaylistStatus::Skipped(file_size)` (new variant) instead of `Done`. Display distinctly on dashboard (e.g., dimmed or with "skipped" label). **Important:** `check_all_done()` in `tui/dashboard.rs` must include `Skipped` in its completion check, or the rip loop will hang waiting for skipped playlists.
- remux.rs: Keep returning `MediaError::OutputExists`. Callers handle it.

**Behavior with `overwrite = true`:**
- CLI: Delete existing file, proceed with rip. Print: "Overwriting {filename} ({size})".
- TUI: Delete existing file, proceed with rip. No special status.
- remux.rs: Remove the `OutputExists` check entirely when overwrite is enabled. Alternatively, handle deletion in the callers before calling `remux()`.

**Implementation:** Callers (CLI and TUI dashboard) handle the overwrite decision before calling `remux()`. The `overwrite` flag is resolved from `--overwrite` CLI flag -> `BLUBACK_OVERWRITE` env -> `config.overwrite` -> default false.

### 5.3 Files

- `src/config.rs`: `overwrite` field, accessor, serialization
- `src/main.rs`: `--overwrite` CLI arg
- `src/cli.rs`: overwrite logic before remux call
- `src/tui/dashboard.rs`: overwrite logic in `start_next_job`
- `src/types.rs`: `PlaylistStatus::Skipped(u64)` variant, settings panel item

### 5.4 Tests

- Config parsing and resolution tests
- (Overwrite logic itself is I/O-dependent â€” test by inspection and manual testing)
